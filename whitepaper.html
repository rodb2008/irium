Technical Whitepaper - Version 1.0.0

MAINNET LIVE Genesis: cbdd1b...000000 Launch: Oct 16, 2025

Abstract

Irium is a purpose-built proof-of-work blockchain designed to maximize network independence and long-term survivability. The protocol eliminates reliance on DNS infrastructure, enforces transparent founder vesting via on-chain timelocks, incentivizes transaction relay quality without inflation, and prioritizes light client usability from genesis.

1. Introduction

Most established proof-of-work networks inherit architectural assumptions from Bitcoin, including DNS-based bootstrapping, addrman-driven peer discovery, and an absence of protocol-level incentives for fast relay. Irium rethinks these components to produce a mainnet that can launch and sustain itself even if all founding infrastructure disappears.

Irium launched on October 16, 2025 with:

  • Mined genesis block (5.4 billion hashes, 7 hours)
  • Zero DNS dependencies
  • Complete P2P networking
  • All services operational

3. The 8 Core Innovations

3.1 Zero-DNS Bootstrap

Problem: DNS is centralized, censorable, and a single point of failure.

Solution: Signed seedlist.txt with raw IP multiaddrs (IPv4 + IPv6), signed anchors.json with checkpoint block headers, bootstrap script with no DNS queries.

✅ Status: Implemented and operational

3.2 Self-Healing Peer Discovery

Problem: Networks need stable, honest peers.

Solution: Uptime proof system (HMAC challenges), peer reputation scoring (0-1000 scale), automatic peer promotion/demotion.

✅ Status: Implemented

3.3 Genesis Vesting with CLTV

Problem: Founder allocations often lack transparency or enforcement.

Solution: 3.5M IRM locked in genesis block with OP_CHECKLOCKTIMEVERIFY. Unlock heights: 52,560 / 105,120 / 157,680 blocks. Consensus-enforced, irreversible.

✅ Status: Implemented in genesis block

3.4 Per-Transaction Relay Rewards

Problem: No incentive to run relay nodes.

Solution: Relay nodes earn 10% of transaction fees. Up to 3 relays per transaction (50%, 30%, 20% split). No supply inflation.

✅ Status: Implemented

3.5 Sybil-Resistant P2P Handshake

Problem: Botnets can saturate networks with fake peers.

Solution: Proof-of-work challenge during handshake, ephemeral key signing, timestamp validation.

✅ Status: Implemented

3.6 Anchor-File Consensus

Problem: Eclipse attacks can feed new nodes false chains.

Solution: Signed checkpoint headers (anchors.json), multiple trusted signers, verification against anchors.

✅ Status: Implemented

3.7 Light Client First (SPV + NiPoPoW)

Problem: Mobile devices can't store full blockchain.

Solution: SPV (Simplified Payment Verification), NiPoPoW (Non-Interactive Proofs of Proof-of-Work), header-only sync, merkle proof verification.

✅ Status: Implemented

3.8 On-chain Metadata Commitments

Problem: No native way to timestamp documents.

Solution: Coinbase metadata field, hash pointers to off-chain data, immutable timestamp proofs.

✅ Status: Implemented
